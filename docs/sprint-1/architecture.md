# Архитектура репозитория

## Цель
Зафиксировать структуру проекта и правила размещения логики для команды.

## Слои и ответственность

### `src/sda/core/generation/`
Доменная логика генерации данных.
1. Генерация строк по шаблонам.
2. Правила связей между сущностями.
3. Без зависимости от FastAPI и HTTP-типов.

### `src/sda/core/anonymization/`
Доменная логика анонимизации.
1. Применение правил к колонкам/значениям.
2. Реализация методов `keep/mask/redact/pseudo`.
3. Без зависимости от web-слоя.

### `src/sda/core/ml/`
Эвристики и синтез похожих данных.
1. `suggest_pii` для подсказок по чувствительным полям.
2. `similar/synthesize` для post-MVP.
3. Без HTTP-контрактов.

### `src/sda/io/`
CSV-ввод/вывод.
1. Чтение CSV.
2. Запись CSV.
3. Без бизнес-решений по анонимизации/генерации.

### `src/sda/use_cases/`
Сценарии прикладного уровня.
1. Оркестрация `core + io + web`.
2. Сквозной workflow фич.

### `src/sda/web/routers/`
HTTP endpoint'ы.
1. Прием запроса и вызов use case.
2. Возврат кодов и ответов.
3. Без тяжелой доменной логики внутри роутеров.

### `src/sda/web/schemas/`
Pydantic-схемы API.
1. Валидация входа/выхода.
2. Контракты endpoint'ов.

### `src/sda/resources/templates/`
Конфигурации шаблонов генерации.

### `tests/`
Unit и API тесты.

## Правила зависимостей
1. `core` ничего не знает о FastAPI.
2. `web` делегирует логику в `use_cases`.
3. `use_cases` связывает слои.
4. `io` занимается только преобразованием CSV.
5. `schemas` не заменяют доменные модели.

## Куда добавлять новую логику
1. `suggest_pii`: `core/ml` -> `use_cases/suggest_pii.py` -> `web/routers/suggest.py`.
2. `similar`: `core/ml/synthesizer.py` -> `use_cases/synthesize_csv.py` -> `web/routers/similar.py`.

## Антипаттерны
1. Дублирование бизнес-логики в роутерах и use case одновременно.
2. Чтение/запись CSV напрямую в роутерах.
3. Протаскивание HTTP-статусов в `core`.
4. Хардкод схем шаблонов в endpoint-коде.
